# Материалы для пояснительной записки

## 1. Как запустить систему

### Локально (разработка)

1. PostgreSQL, создать БД `inventory_db`.
2. `cd backend && cp .env.example .env` — заполнить `DATABASE_URL`, `JWT_SECRET`.
3. `go run ./backend` из корня репозитория (где `go.mod`) или `cd backend && go run .`
4. Фронт: `cd frontend && npm install && npm run dev`.

### Docker Compose (одна команда для проверяющего)

```bash
docker compose up --build
```

API: `http://localhost:8080`, health: `GET /health` (проверка БД: поле `database`).

## 2. Схема REST API (кратко)

| Метод | Путь | Назначение |
|--------|------|------------|
| POST | `/auth/login` | Вход, JWT |
| GET | `/auth/me` | Текущий пользователь (Bearer) |
| GET | `/items` | Список, `?q=` поиск |
| GET | `/categories` | Справочник категорий объектов учёта |
| POST | `/categories` | Новая категория (только **admin**) |
| GET | `/items/export` | **CSV экспорт** учётных объектов |
| CRUD | `/items`, `/items/:id` | Объекты (`category_id` — связь со справочником) |
| GET | `/inventory/sessions/:id/stock-ledger` | Реестр списаний/оприходований по проведённому документу |
| — | `/inventory/sessions` … | Сессии (документы) инвентаризации и строки опись |

Полный список — в корневом `README.md`.

## 3. Отличительные возможности (уровень выше «простого CRUD»)

- **JWT + роли** (admin / user): изменение учётных объектов и экспорт CSV только у **admin**; **user** читает справочник и ведёт свои сессии инвентаризации; **admin** видит все сессии.
- **Справочник категорий** (отдельная таблица): выбор категории из списка в форме объекта, фильтр при создании документа инвентаризации, ведение категорий в настройках (admin).
- **CORS** для SPA (Vite) и переменная `CORS_ORIGINS`.
- **Health** с проверкой доступности PostgreSQL (`/health`).
- **Экспорт CSV** по учётным объектам (`GET /items/export`) — для отчётности (в т.ч. `category_id` и название категории).
- **Инвентаризация как упрощённый документ** (аналог логики 1С без полного комплекта первичных документов):
  - создание документа с опциональной выборкой объектов (локация, категория или все не списанные);
  - в каждой строке — **учётное** количество (снимок на момент создания документа) и **фактическое** (реальное), ввод после сканирования QR;
  - действие **«Провести документ»**: выравнивание остатков в карточках под факт; фиксация **реестра движений** (списание/оприходование) с привязкой к сессии как к документу-регистратору; дублирование смысла в **истории объекта**;
  - непосчитанные строки остаток в карточке не меняют (при строгом завершении их нельзя оставить; админ может `allow_incomplete`).
- **Минимальный остаток** — не для инвентаризации, а для **контрольного списка** «мало на складе» и отчётов.
- **Замена объекта**: при списании указывается UUID преемника; в карточке отображается, **на что заменено** (SKU и название преемника).
- **QR-код**: кодирует URL веб-карточки; изображение строится на клиенте библиотекой `qrcode.react` (стандарт QR). Отдельного хранения файла на сервере нет; при создании объекта ссылка уже известна, код отображается сразу.
- **Фронт**: единый разбор ошибок API, единица по умолчанию **«шт»**, нормативный срок по умолчанию **4 года** (как в бизнес-правилах клампа на backend).
- **Docker Compose** + **CI** (`go test` в GitHub Actions).
- **Тесты** на конфиг, middleware, бизнес-правила статусов.

## 5. Теоретические вставки (черновик для ВКР)

Ниже — готовые фрагменты, которые можно перенести в главы теоретической части и при необходимости стилистически доработать под требования кафедры.

### 5.1 Требования (для п. 1.1)

**Функциональные требования:**
- аутентификация пользователей с разграничением ролей `admin/user`;
- ведение справочника объектов учета (создание, изменение, удаление, поиск);
- проведение инвентаризационных сессий с фиксацией фактических значений;
- вычисление статусов расхождений (`match`, `mismatch`, `missing`);
- формирование выгрузки данных в формате CSV;
- предоставление интерфейса для просмотра сессий и их результатов.

**Нефункциональные требования:**
- клиент-серверная архитектура с централизованным хранением данных в PostgreSQL;
- устойчивость к типовым ошибкам ввода и отказам внешних компонентов;
- безопасность доступа на основе JWT и ролевой модели;
- адаптивность пользовательского интерфейса для мобильных устройств;
- сопровождаемость за счет модульной структуры backend/frontend.

### 5.2 Обоснование выбора технологий (для п. 1.2)

Выбор стека `Go + Gin + GORM + PostgreSQL` на серверной стороне обусловлен сочетанием высокой производительности, простоты развертывания и удобства реализации REST API. Язык Go обеспечивает эффективную обработку конкурентных запросов, а фреймворк Gin позволяет быстро организовать маршрутизацию, middleware и валидацию входных данных. Использование GORM упрощает работу со схемой данных и снижает трудоемкость операций CRUD при сохранении прозрачности SQL-взаимодействия.

На клиентской стороне использован стек `React + TypeScript + Vite`, обеспечивающий компонентный подход к разработке интерфейса, типобезопасность и высокую скорость сборки в режиме разработки. Для обмена данными применяется HTTP-клиент Axios, а управление маршрутизацией реализовано через React Router. В совокупности выбранный технологический набор обеспечивает баланс между скоростью разработки, расширяемостью и удобством сопровождения.

### 5.3 Архитектурная модель (для п. 1.3)

Приложение реализовано по клиент-серверной модели. Клиентская часть формирует HTTP-запросы к REST API серверной части и отображает результаты обработки в пользовательском интерфейсе. Серверная часть выполняет аутентификацию, проверку прав доступа, обработку бизнес-логики инвентаризации и взаимодействие с базой данных. Между слоями соблюдается четкое разделение ответственности, что позволяет независимо развивать интерфейс и backend-компоненты.

В качестве формата взаимодействия выбран REST-подход с передачей данных в JSON. Доступ к защищенным маршрутам осуществляется по JWT-токену в заголовке `Authorization`. Такая модель обеспечивает простую интеграцию с веб-клиентом и позволяет масштабировать систему за счет разделения вычислительной нагрузки между компонентами.

### 5.4 Безопасность (для пп. 3.1–3.3)

Ключевые угрозы включают несанкционированный доступ к API, подбор учетных данных, компрометацию токена и некорректную обработку входных данных. Для снижения рисков реализованы аутентификация по JWT, хранение паролей в виде bcrypt-хэшей, ролевое разграничение операций и CORS-политика для клиентских приложений.

Безопасность хранения данных обеспечивается централизованным размещением информации в PostgreSQL и ограничением прямого доступа к базе через серверный слой. Безопасность передачи данных в эксплуатационном контуре должна обеспечиваться использованием HTTPS. Для дальнейшего усиления защиты рекомендуется добавить ограничение частоты запросов на маршрут авторизации и строгую политику управления секретами окружения.
